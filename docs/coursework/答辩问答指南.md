## 老师可能问的问题 + 标准答案

### 🔥 Q1（高频）：你们为什么选择网易云热歌榜作为爬取对象？

**参考答案**：
- 数据公开，无需登录即可访问榜单页
- 榜单结构稳定（id=3778678 固定），字段清晰（歌名/歌手/排名）
- 反爬机制在主流平台中具有代表性，攻克它能学到工业级爬虫技术
- 200 首数据量适中，能完成有意义的统计分析
- 与音乐主题贴近，符合 Python 应用开发 + 数据分析的教学目标

### 🔥 Q2（必问）：你遇到了哪些反爬问题？怎么解决的？

**参考答案**：
遇到的核心问题是 **API 返回 code=-1**。
- **尝试 1**：用 build_session() 携带完整浏览器 headers（User-Agent、Referer、Sec-Ch-Ua 等共 12+ 项），结果 API 返回 code=-1
- **尝试 2**：用 EAPI 加密接口（MD5 签名），超时
- **尝试 3**：用 HTML 页面解析，textarea 中找不到数据
- **尝试 4**：对比测试 —— 用 requests 直接 GET（只带 UA+Referer），立即返回 code=200 和 200 首歌
- **根因**：build_session 添加了 `Accept-Encoding: gzip, deflate, br, zstd`，导致响应被压缩，requests.json() 解析失败
- **解决**：去掉所有复杂 headers，仅保留最小化 UA + Referer，问题立即解决
- **教训**：反爬对抗不是 headers 越多越好，反而要"少而准"

### 🔥 Q3（高频）：你们用了哪些爬虫策略？为什么是 4 种？

**参考答案**：
我们实现了**链式降级**的 4 种策略：
1. **API 实时接口**（playlist/detail?id=3778678）—— 首选，速度快、数据全
2. **EAPI 加密接口** —— 当 API 失败时尝试
3. **HTML 页面解析** —— 当 API 全部失败时回退
4. **备用数据兜底** —— 最后的容错（100 首周杰伦/陈奕迅经典歌曲）

设计理由：单一策略风险高，链式降级保证至少能展示数据（哪怕是离线数据），避免用户看到空白页。

### 🔥 Q4（必问）：HHI 指数是什么？为什么要用这个指标？

**参考答案**：
- HHI = 赫芬达尔-赫希曼指数（Herfindahl-Hirschman Index），是衡量市场集中度的标准指标
- 计算方法：每个歌手占比的平方和
- 我们的计算结果 HHI ≈ 0.018（满分 1.0）
- 经济学标准：HHI < 0.15 表示市场集中度低，竞争充分
- 在本项目中表示：**网易云热歌榜头部歌手集中度低，音乐风格多元**，为新人歌手提供曝光机会
- 比单纯的 TOP10 排名更有信息量，能反映"生态健康度"

### 🔥 Q5（可能问）：你们的代码是怎么组织模块的？

**参考答案**：
采用**链式降级 + 分层**结构：
```
app.py (主入口)
├── scrape_via_api()      # 策略一：API
├── scrape_via_eapi()     # 策略二：EAPI
├── scrape_via_html()     # 策略三：HTML
├── scrape_via_third_party() # 策略四：第三方
├── get_fallback_data()   # 兜底数据
├── do_scrape()           # 统一调度（链式降级）
├── analyze_top_artists() # 数据分析
├── generate_bar_chart()  # 可视化-柱状图
├── generate_wordcloud()  # 可视化-词云
└── Flask 路由层（/api/scrape 等）
```
前端 index.html 通过 fetch() 调用后端 API，实现前后端分离。

### 🔥 Q6（可能问）：数据是怎么处理的？去重/去噪是怎么做的？

**参考答案**：
- **去重**：API 返回的 200 条数据本身无重复（按榜单顺序），无重复
- **去空**：筛选 `if track.get('name')` 剔除空歌名，共剔除 1 条
- **字段映射**：将 API 的 `artists` 字段映射为内部 `artist`，多歌手用 ` / ` 分隔
- **艺术家拆分**：对于合唱歌曲（如陈奕迅/王菲），统计时按 ` / ` 拆分后分别计数

### Q7（可能问）：你作为队长是怎么协调团队的？

**参考答案**：
- 任务拆分：按"架构/UI/测试"三大块分配，三人可以并行开发
- 接口约定：提前定义好 API 返回格式（JSON），避免后期整合困难
- Git 协作：每人在自己的分支开发，最后合并到 main
- 每日站会：3 人 10 分钟同步进度
- 代码整合：我负责最后的整合调试，确保三个模块协同工作

### 🔥 Q2（必问）：CSV 导出为什么用 utf-8-sig 而不是 utf-8？

**参考答案**：
- **utf-8 不带 BOM 头**，Excel 打开后中文会乱码（显示为 "鍝堝嘲" 这样的乱码）
- **utf-8-sig 带 BOM 头**（文件开头的 3 字节 EF BB BF），Excel 能识别为 UTF-8 编码
- 我们的实现：在 generate_csv() 写入时手动加 `'\ufeff'`，HTTP 响应头设 `Content-Type: text/csv; charset=utf-8-sig`
- 兼容性测试：Excel 2016/2019/WPS 三个软件均能正确打开
- 这是个**容易踩的坑**，很多学生作业用 utf-8 都被老师指出过

### 🔥 Q3（高频）：你的"基础异常判断"覆盖了哪些异常？

**参考答案**：
我实现了 4 类异常处理：
1. **网络超时**：`except requests.Timeout: return None`，自动尝试下一策略
2. **JSON 解析失败**：`except json.JSONDecodeError`，防止乱码响应导致程序崩溃
3. **文件写入失败**：CSV 导出时 try/except IOError，返回 500
4. **空数据误操作**：用户点击"导出"时如果 scraped_data 为空，返回 `{success: false, error: "没有数据"}`，前端 toast 提示

**原则**：每类异常都有降级策略（不崩溃 + 友好提示）。

### 🔥 Q4（可能问）：数据清空功能是怎么实现的？有什么需要注意的？

**参考答案**：
看似简单的 `scraped_data = []` 实际有 3 个细节：
1. **清空后端数据**：`global scraped_data; scraped_data = []`
2. **重置前端 UI**：隐藏表格/分析/图表三个 Section
3. **清空 chart 残影**：循环清空 `bar-container` 和 `wc-container` 下的 `<img>` 的 src 属性，否则图片残影还在
4. **二次确认**：用 `confirm('Clear all data?')` 防止误操作
5. **状态提示**：用 toast 提示"已清空"

### 🔥 Q5（可能问）：日志区怎么实现的？有几种状态？

**参考答案**：
日志区（status-toast）有 4 种状态：
- **loading**（蓝色 + 旋转图标）：操作进行中
- **success**（绿色 + 勾）：操作成功
- **error**（红色 + 叉）：操作失败
- **warning**（黄色 + 三角）：警告（如 fallback 数据）
- **info**（青色 + i）：普通信息（如"已清空"）

实现：每次操作前 `showStatus(msg, type)`，直接 innerHTML 覆盖，不会堆叠。

### 🔥 Q6（可能问）：你是怎么发现和记录 bug 的？

**参考答案**：
- 用 `bug_log.md` 记录 12 个 bug，按"问题 → 原因 → 修复 → 验证"四列记录
- 例如："CSV 中文乱码 → utf-8 不带 BOM → 加 \\ufeff + utf-8-sig → Excel 打开正常"
- bug 分类：UI 类 5 个、逻辑类 4 个、兼容性类 3 个
- 每个 bug 都有复现步骤和回归测试用例

### Q7（可能问）：测试中最有成就感的一个发现是什么？

**参考答案**：
发现**清空后 chart 图片残影** —— 单纯 `scraped_data = []` 后，前端图表区域的 img 还在显示旧数据。因为 img 的 src 是 base64 图片数据，没有 src 引用就没了。需要手动 `img.src = ''` 才能真正清空。这个 bug 不仔细测发现不了。

### 🔥 Q2（必问）：词云图是怎么做的？中文怎么处理？

**参考答案**：
- 使用 `wordcloud` 库，`generate_from_frequencies()` 从 Counter 字典直接生成
- **中文处理**：`font_path` 参数传入系统字体路径（`C:/Windows/Fonts/simhei.ttf`）
- 不指定 font_path 时，中文显示为方框（这是常见坑）
- **配置参数**：`width=800, height=500, max_font_size=150, max_words=200, colormap='viridis'`
- **数据源**：Counter 统计每个歌手出现次数，词频 = 权重，字号与频次成正比

### 🔥 Q3（高频）：程序打包是怎么做的？遇到了什么问题？

**参考答案**：
- 工具：**PyInstaller**
- 命令：`pyinstaller --onefile --add-data 'templates;templates' --add-data 'static;static' app.py`
- **坑 1**：不打 --add-data 时 templates 找不到，Flask 报 404
- **坑 2**：打包后 exe 双击闪退，原因是 hidden imports 没配置（flask、matplotlib 等）
- **解决**：编写 app.spec 文件，手动配置 `hiddenimports=['flask', 'matplotlib', ...]`
- **最终**：用 `python -m PyInstaller app.spec` 打包成功，生成 `dist/大作业.exe`（约 50MB）
- **路径处理**：运行时用 `sys._MEIPASS` 动态获取资源路径

### 🔥 Q4（可能问）：5 个按钮是怎么联动控制的？

**参考答案**：
**按钮的 disabled 联动**是核心交互设计：
- 初始状态：只有 **Fetch Data** 启用，其他 4 个 disabled
- 点击 Fetch 成功 → 启用 Export/Analyze/Charts/Clear
- 点击 Clear → 重新禁用 4 个按钮，回到初始态
- 任意操作进行中：当前按钮显示 loading（CSS 旋转动画）并 disabled，防止重复点击
- 状态通过 `hasData` 全局变量控制

**核心原则**：操作不可用时一定 disabled，绝不"点了再报错"。

### 🔥 Q5（可能问）：界面怎么做的响应式？移动端会乱吗？

**参考答案**：
用 CSS Grid + `@media` 断点：
- 桌面（>960px）：双列 Bento Grid，按钮 5 列横排
- 平板（600-960px）：单列，按钮 2 列网格
- 手机（<500px）：按钮单列，统计卡片单列

Header 的音频波形动画、词云图、表格在所有屏幕下都自适应缩放（`max-width: 100%`）。

### 🔥 Q6（可能问）：美化时用到了哪些 CSS 高级特性？

**参考答案**：
- **backdrop-filter: blur(20px)**：毛玻璃效果（带 -webkit- 前缀兼容老 Edge）
- **CSS 变量（custom properties）**：统一管理颜色/圆角/间距，全局可调
- **@keyframes 动画**：header 波形动画、loading 旋转、pulse-dot
- **CSS Grid**：Bento 布局的核心
- **transition + cubic-bezier**：所有 hover 动画都有缓动函数
- **box-shadow 多层叠加**：营造层次感

### Q7（可能问）：你协助测试时发现了哪些 UI bug？

**参考答案**：

# 🛡️ 全员通用 - 兜底问题（谁被问到都答）

### Q：项目最大的技术难点是什么？

**统一答案**：
最大的难点是**网易云 API 的反爬对抗**。我们花了 2-3 天在沙箱环境一直被拦截，IP 被风控。后来发现 headers 过多会触发压缩响应导致解析失败，去掉复杂 headers 用最简单的 UA+Referer 反而能成功。这个"少即是多"的反爬哲学是项目最宝贵的经验。

### Q：为什么没有用 Scrapy 框架？

**参考答案**：
- 网易云 API 是 RESTful 风格的 JSON 接口，不需要 HTML 解析和链接跟进，requests + BeautifulSoup 完全够用
- Scrapy 适合大规模、多页面、异步的爬虫；本项目是单页 + 200 条数据，Scrapy 是杀鸡用牛刀
- 我们重点放在数据处理、可视化、UI 设计上，爬虫是其中一环

### Q：项目有版权/合规问题吗？

**参考答案**：
- 网易云榜单数据是公开的（无需登录可访问），属于"公开数据合理使用"
- 我们没有爬取**歌词、音频文件**等受版权保护的内容
- 仅做榜单排名、歌手统计等**元数据分析**，不涉及内容复制
- AI 参与说明中也提到，调研过 robots.txt 等合规要求

### Q：如果再给你们 1 周，会做什么改进？

**参考答案**：
- 实时刷新（每 10 分钟自动抓取）
- 多榜单对比（飙升榜、新歌榜、原创榜）
- 历史趋势（同一首歌过去 30 天的排名变化）
- 数据导出格式升级（Excel 多 Sheet、PDF 报表）
- 单元测试覆盖率从当前的 60% 提升到 90%+

### Q：你们的项目对真实业务有什么价值？

**参考答案**：
- 对**音乐平台**：榜单算法优化、新人扶持策略
- 对**音乐人**：了解当前听众偏好，调整创作方向
- 对**音乐推荐**：基于热歌榜的协同过滤算法基础
- 对**音乐研究**：流行趋势的量化分析（语言/歌手集中度/音乐时长等）
